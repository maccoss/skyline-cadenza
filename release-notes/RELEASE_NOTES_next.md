# Skyline Cadenza Release Notes (Next Release)

Working draft for the next release. Append entries here as features and fixes land on `main`. At release time this file is renamed to `RELEASE_NOTES_v{version}.md` and `Version` in `src/SkylineCadenza.App/tool-inf/info.properties` is updated to match.

## New Features

- **Push-to-Skyline now writes a BiblioSpec library, configures the document settings, and populates the target tree in one click.** The flow is four stages, all driven through the JSON-RPC pipe â€” exclusively via `RunCommand`, which per Nick Shulman behaves as if the same `SkylineCmd`-style commands were typed into Skyline's Immediate Window:

    1. Write a self-contained `.blib` next to the active document (one reference spectrum per scheduled peptide, top-6 library fragments with m/z + intensity + charge, per-replicate `RetentionTimes` row carrying the firing window).
    2. Register the library with `--add-library-name=NAME --add-library-path=PATH`. This calls Skyline's own `CommandLine.SetLibrary` â†’ `LibrarySpec.CreateFromPath`, which dispatches on the `.blib` extension to construct a `BiblioSpecLiteSpec` and appends it to `PeptideSettings.Libraries.LibrarySpecs` â€” registered AND active in one shot. We attempted earlier to register via `AddSettingsListItem("Spectral Libraries", xml)`; that path is broken in Skyline's `JsonToolServer` because the deserializer looks up `Deserialize(XmlReader)` on the generic argument of `SettingsListBase<T>`, which for spectral libraries is the abstract `LibrarySpec` (no static `Deserialize`). Replaced entirely.
    3. Configure the document peptide / transition / full-scan filter via a sequence of one-flag `RunCommand` calls derived from the assay shape: `--pep-min-length` / `--pep-max-length` from peptide-length extremes; `--tran-precursor-ion-charges` / `--tran-product-ion-charges` from observed charge unions; `--tran-product-ion-types=y,b`; `--tran-product-start-ion=ion 1`; `--tran-product-end-ion=last ion`; `--library-pick-product-ions=filter`; `--library-product-ions=6` (matches the BLIB writer's peaks-per-spectrum); `--full-scan-rt-filter=scheduling_windows` with `--full-scan-rt-filter-tolerance=X` where X is the assay's worst-case peak half-width plus `FiringPadSec / 60`, rounded up to 0.1 min. For MTM mode an additional `--full-scan-isolation-scheme=Results only --full-scan-acquisition-method=DIA` is bundled into a single `RunCommand` (Skyline's `TransitionFullScan.DoValidate` enforces the mutual lockout â€” `AcquisitionMethod = DIA` requires a non-null isolation scheme, `AcquisitionMethod = None` requires a null one, so neither flag passes alone from a fresh document; one combined call validates only the destination state). For PRM mode the acquisition method is just `--full-scan-acquisition-method=PRM` with no isolation scheme. One flag per batch elsewhere so a future unknown-flag error only loses its own setting.
    4. Import a peptide-style transition list via `--import-transition-list=PATH`. Each row carries the real `Product Charge` from `FragmentIon.Charge`, replacing the previous hardcoded `+1` that was the source of "no matching product ion" errors against fresh documents. `Candidate.Fragments` now carries up to 12 fragments (m/z + intensity + charge) from each of the three ingest paths (DIA-NN report's `Fr.x.Id` annotations, Carafe TSV's `FragmentCharge` column, Skyline document's `ProductCharge`).

  Two settings still require a one-time manual UI step in **Peptide Settings â†’ Prediction** because no SkylineCmd flag currently exposes them: set the *Alignment target* to "Library RTs" (so run-to-run alignment uses the BLIB-sourced library retention times) and check *Impute boundaries for missing peaks*. Both persist across Skyline sessions.
- **BLIB-sourced firing windows for "Load from Skyline document".** Cadenza now reads per-peptide measured peak boundaries directly from the BiblioSpec `.blib` files the active Skyline document references, via `GetSettingsListSelectedItems("Libraries")` + `GetSettingsListItem` and direct SQLite read of the BLIB's `RetentionTimes` table. The firing window is the union (`MIN(startTime)`, `MAX(endTime)`) across every replicate that detected the peptide, filtered to observations whose identification confidence satisfies the same `QValueCutoff` used at ingest. When a document references multiple BLIBs (e.g. a measured DIA-NN library plus a predicted Carafe / Prosit library), each library is scored by per-replicate boundary variance and the higher-scoring library wins per peptide â€” so DIA-NN naturally outranks predicted libraries without Cadenza having to inspect their LSIDs. Falls back to `RT Â± peakHalfWidthMin` synthesis for peptides not covered by any BLIB. Status line on a successful load now reports "N of M firing windows from BLIB".
- **Coverage objective selector simplified to a single radio group**. The scheduling panel now offers three options under "Coverage objective" with no separate load-balancing / look-ahead checkboxes: *Balanced* (the exact published Stellar-webinar algorithm: reactive cover + round-robin load-up to `MaxPeptidesPerProtein`), *Maximize Protein Coverage* (default: look-ahead cover with prefer-joinable + load-up capped at `MinPeptidesPerProtein` so saved budget feeds more first-peptide protein coverage), and *Maximize Peptide Coverage* (look-ahead cover + uncapped load-up, round-robin fairness preserved). `SchedulingParameters.EnableLoadBalancing` and `RtAwareCoverSelection` are removed; the objective alone drives both cover-pass strategy and load-up cap. The default flips from the previous "Balanced + look-ahead" (look-ahead cover + round-robin to Max) to "Maximize Protein Coverage" (look-ahead cover with joinable-first + cap at Min), matching what most users want from the default.

- **Headless `SkylineCadenza.Cli` for scripting and analysis.** New `src/SkylineCadenza.Cli/` console project takes a `ScheduleRequest` JSON (candidates + `SchedulingParameters`) on stdin or a file, runs the real C# `Scheduler.Run`, and emits a `ScheduleResponse` JSON (indices, slot ids, slots, RT grid, slot count curve, protein groups covered). Invoke as `dotnet run --project src/SkylineCadenza.Cli -- schedule INPUT.json OUTPUT.json` (use `-` for stdin/stdout). Wire-format DTOs are kept separate from the Core types so the on-disk schema is stable across Core refactors; enums round-trip as strings.

- **Algorithm-comparison Jupyter notebook with rendered figures.** `notebooks/algorithm-comparison.ipynb` walks through how the three coverage objectives differ, using a Python re-implementation for a five-protein toy walkthrough and the real C# scheduler (via `SkylineCadenza.Cli`) for ~150-protein realistic plots: RT slot-occupancy curves, four-panel outcome summary, and a per-protein peptide-count heatmap. Generator at `notebooks/_build_notebook.py`, re-render workflow documented in `notebooks/README.md`. Figures are committed in `notebooks/figures/` so they render on GitHub without re-execution.

- **`uv`-managed Python environment at the repo root.** New `pyproject.toml` declares the Python dependencies for the notebook and tooling (matplotlib, numpy, jupyter, nbconvert, nbformat, ipykernel, debugpy). `uv.lock` is committed for reproducible installs. Visual Studio Code's Python + Jupyter extensions auto-detect the workspace `.venv/` as the kernel for the notebook; setup is `uv sync` from the repo root. `notebooks/README.md` documents the workflow.

## Bug Fixes

- **Protein-coverage plot title now reports covered-of-total.** Previously the title showed only the total group count (`n = 7,740 groups`), which obscured the fact that the "Not scheduled" greys are visually overdrawn by adjacent colored markers at high density. The title now reads `(6,634 of 7,740 groups covered)` so the visible curve and the title agree on what is actually distinguishable.
- **Protein-coverage "5+ peptides" legend entry now reads "5 peptides" when no protein exceeds 5.** Under the default Balanced objective with `MaxPeptidesPerProtein = 5` nothing can reach 6, so the previous "5+" wording was misleading. The legend is dynamic and reverts to "5+ peptides" under the new MaximizePeptides objective when the load-up actually exceeds 5.
- **"Load from Skyline document" button no longer carries the (preview) tag.** Precursor / fragment ingest from a running Skyline document is now backed end-to-end: report probe for the spectral data, BLIB read for the per-replicate firing windows (see the New Features entry above).
- **MTM slot edges now reserve room for the solo quadrupole window.** Every member peptide's quadrupole window (`PrmIsolationWidthTh` wide, default 0.7 Th, centered on the precursor m/z) must fit fully inside the slot's nominal isolation window. Cadenza now requires every peptide center to sit at least `PrmIsolationWidthTh / 2` from each slot edge. Previously, peptide centers were allowed to span the full `IsolationWindowTh`, leaving precursors at the extreme of a slot with half their quadrupole window outside the nominal slot edge. For the default 3.0 / 0.7 pair this caps the span of precursor centers at 2.3 Th; for any other window / solo pair the rule scales naturally. The Thermo CSV writer now reports `memberSpan + PrmIsolationWidthTh` as the instrument isolation width so the actual instrument firing covers what the scheduler reserved.

## Breaking Changes

- **`SchedulingParameters.EnableLoadBalancing` and `RtAwareCoverSelection` are removed.** Both are now folded into the `CoverageObjective` enum; the objective alone determines cover-pass strategy and load-up cap. Code that set these explicitly should pick an objective instead: `Balanced` corresponds to the old `EnableLoadBalancing = true, RtAwareCoverSelection = false`; `MaximizeProteins` (default) corresponds to "look-ahead with prefer-joinable + load-up only up to Min"; `MaximizePeptides` corresponds to "look-ahead with uncapped load-up". The previous default combination (look-ahead + round-robin to Max) is no longer a single radio option â€” pick `MaximizePeptides` and let load-up run if you want the closest equivalent.
- **`SchedulingParameters.Objective` default is now `MaximizeProteins`, not `Balanced`.** This is a stricter default (cover-only with Min = 1 by default, no load-up) but covers more proteins on tight-budget assays. Set `Objective = CoverageObjective.Balanced` to recover the published-webinar reactive-cover + round-robin-load-up-to-Max behaviour. The previous "Balanced + look-ahead" combination (what was the actual default) is no longer reachable in one knob.
- **`Top4Cache` parquet schema bumped: `FragmentIntensity` and `FragmentCharge` columns are now required.** Cached top-N parquets written by previous Cadenza or notebook versions will fail to load with a missing-column error and rebuild automatically on the next ingest via the existing SHA-1 cache key. No data loss â€” just a one-time rebuild cost the first time a Carafe ingest runs against an existing cache directory.
- **`Candidate.Top4Fragments` is still present as a derived view, but `Candidate.Fragments` (a `FragmentIon[]`) is the new source of truth.** Code that constructed `Candidate` directly with `Top4Fragments = ...` will fail to compile â€” set `Fragments = ...` (a `FragmentIon[]`) and `Top4Fragments = Candidate.DeriveTopMz(fragments, 4)`. Tests using the `SchedulerTests.Make` helper see this transparently; external consumers building Candidates need to update.
- **`PeptideTransitionListBuilder` and `TransitionListBuilder` moved from `src/SkylineCadenza.Core/SkylineRpc/` to `src/SkylineCadenza.Core/Output/`** alongside the new `BlibAssayWriter`. `TransitionListBuilder` (small-molecule layout) was unused after the proteomic switch and is deleted; `PeptideTransitionListBuilder` is preserved and emits real `FragmentIon.Charge` values per fragment.
